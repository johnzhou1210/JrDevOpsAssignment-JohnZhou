Task #5
    1. To version the application, I was thinking of just passing in an input to the workflow to take in a version string. However, since Task #5 wants me to only run the workflow upon pushing to main, I can no longer use workflow inputs for app name and version.
        - Instead of passing app name and version via manually running the workflow, I can create a file that the .yaml script can read from to build with a custom name and version.
        - I created an app.config file in repo's root which contains information about the app's name and version. By doing this, the .yaml script will be able to automatically take in the inputs app name and version upon building.
    2. One issue I encountered was with a syntax gotcha when there are empty spaces in a string. For empty spaces, wrapping the variable in double quotation marks treats the variable as a single argument instead of two.
        - Example: 
            APP_NAME="A Test App" 
            ${APP_NAME} -> 3 arguments
            "${APP_NAME}" -> 1 argument
    3. Before looking at the other part of this task, I tried creating a branch off of main called dev to make sure that pushing to this new branch won't trigger the build workflow.
        - Afterwards pushing to dev, merging dev into main indeed triggers the build workflow.
    4. Since this task suggested that the build should not always be deployed/uploaded after a build, my first instinct was to check on the CDN if the version exists there. However, since that isn't an option for this assignment since everything is simulated,
        One other approach is to only upload the artifacts if the app version in the latest commit is newer than the app version in the previous commit. However, for simplicity's sake, I decided to just use a string comparison for version checking, and only upload if
        the previous commit's app version is different from the current's app version.
        - During this step, I encountered an issue where the previous version was defaulting to an empty string. I used AI assistance to help me diagnose the issue. The issue was that git checkout only checked out a shallow clone, and HEAD~1 didn't exist inside the workflow runner.
        - I needed to set the fetch depth to 0 during the Checkout repository step. Its default value is 1, which means that only the latest commit is fetched, which explains why git show couldn't retrieve the previous commit (HEAD~1).
